如果你担心解析性能问题：100 个城市、6000 多行 MD，_parse() 的纯正则解析在首次调用时大概几十毫秒级别，不构成瓶颈。
2.2 city_overview chunk 是 section 过滤失效的元凶
这是我在代码中发现的、你们之前讨论里没有触及的关键问题。init_knowledge_base.py 的 parse_china_tourist_kb() 函数为每个城市生成了两类 chunk：一类是 chunk_type: "section"（按 ### 切分，粒度合理），另一类是 chunk_type: "city_overview"（整个城市的完整内容作为单个 chunk）。
问题在于：即使你给 RAG 检索加上 section metadata 过滤（比如只检索 section: "避坑指南"），city_overview chunk 不带 section 字段，它不会被 section 过滤命中，但如果你只做 city 过滤而不做 section 过滤（当前代码就是这样），这个巨大的 overview chunk 会高概率被召回，因为它包含所有关键词，语义相似度天然就高。一旦召回，LLM 从中抽取时就会看到全部内容，造成内容串台。
建议：重建向量库时，去掉 city_overview chunk 类型，只保留 section 级别的 chunk。city_overview 这种 catch-all 设计在你们的架构里没有明确的消费方，反而是噪声源。如果确实需要"给我这个城市的综合概览"的能力，直接走 CityKnowledgeDB 结构化查表就行，不需要过 RAG 链路。
2.3 rag_experience_node 和 rag_risk_node 的 RAG 必要性值得重新审视
读完 rag_risk/script/agent.py 后我产生了一个更根本的疑问：对于你自己编写和维护的知识库来说，RAG 链路（向量检索 + LLM 抽取）在很多场景下是不是在做无用功？
以 rag_risk_node 为例，它的工作流是：构造 query "杭州 避坑 踩雷 注意事项 常见误区" -> 向量检索 -> 召回若干 chunk -> LLM 从中抽取 risks 列表。但 CityKnowledgeDB.get_tips("杭州") 已经能直接返回避坑指南的原文列表，零延迟、零 token 消耗、零幻觉风险。
真正需要 RAG 的场景是：知识库里存在大量非结构化的叙述性内容，无法通过确定性方式提取。在你的 MD 文件里，避坑指南 本身就是 -  列表条目，CityKnowledgeDB 用正则就能精确抽取。rag_risk_node 经过 RAG + LLM 二次加工反而引入了风险——LLM 可能过度概括或把其他 section 的内容混入。
我的建议是区分两种数据消费模式：

确定性数据（避坑指南、美食推荐、交通枢纽、核心景点种子）：直接走 CityKnowledgeDB 查表，完全不过 RAG。这些数据的格式在你 MD 里已经是规整的列表/条目，正则提取的准确率 > LLM 抽取。
需要语义理解的数据（"这个城市适合情侣游的理由是什么"、"哪些景点适合带小孩"）：才走 RAG，因为这类问题的答案散布在多个 section 中，需要跨段落语义聚合。

如果采纳这个思路，rag_experience_node 的职责可以收窄为：只负责回答风格适配性问题（best_for 字段），以及从知识库中提取那些不在任何结构化列表里但对用户有价值的"隐性知识"。tips 字段则改为从 CityKnowledgeDB 直接获取，不再经过 RAG 链路。
2.4 section 过滤的落地：当前 metadata 已就绪，只差查询时启用
好消息是 init_knowledge_base.py 已经在每个 section chunk 的 metadata 里写入了 "section": "避坑指南" / "section": "本地美食" 等字段。所以实现 section 过滤不需要重建索引结构，只需在 RAGBaseAgent.search_knowledge() 里扩展 where 条件：
pythonif city_filter and section_filter:
    query_kwargs["where"] = {
        "$and": [
            {"city": city_filter},
            {"section": section_filter}
        ]
    }
然后在 RAGExperienceAgent.run() 和 RAGRiskAgent.run() 里传入对应的 section_filter 参数即可。这是改动最小、收益最大的一步。
2.5 accommodation_node 的 P4 自检索方案是正确的
你们讨论中提到 "accommodation_node（P4）自己检索 accommodation section，不依赖 P2 残留"，我完全同意。当前代码里 accommodation_node 已经在做 knowledge_db.get_accommodation(kb_city)，这说明这条路径事实上已经落地了。唯一需要确认的是：orchestrate_node 的 P2 阶段是否还在为 accommodation 做 RAG 检索？如果是的话应该移除，避免做无用功。

## 问题：住宿节点是否应该引入 RAG？
你观察到 accommodation_node 的核心逻辑是计算当天景点坐标重心、调用 Amap MCP 搜周边酒店、再用 RollingGo 补价格，本质是一条纯 MCP 实时数据驱动的链路。你问：在这条链路里引入 RAG 是否反而引入噪声？是否应该像 poi_enrich_node 那样只做补充解释？
结论：不该引入 RAG，而且当前知识库数据实际上是死代码。
我读完代码后发现：accommodation_node.py 已经调用了 CityKnowledgeDB.get_accommodation(city) 并把结果作为 knowledge_accommodation 传给了 AccommodationAgent，但 AccommodationAgent.run() 从头到尾没有读取这个参数。知识库数据传了但没人消费。
不建议引入 RAG 的三个理由：

信息粒度冲突 -- 知识库写的是"上城区湖滨适合住"这种区域级定性描述，MCP 返回的是具体酒店名+价格+距离。两种粒度混给 LLM 会增加幻觉风险。
破坏白名单约束 -- prompt 里已经花大力气做了"酒店名必须来自 MCP 白名单"的约束。RAG 召回的住宿指南如果提到了 MCP 列表之外的酒店/民宿名称，会诱导 LLM 突破白名单。
检索时机和语义都不合适 -- 住宿指南的价值是"哪个区域值得住"的战略判断，不是"哪家酒店好"的具体推荐。这种先验知识适合作为 prompt 静态上下文注入，不适合走向量检索链路。

建议方案：静态上下文注入，知识库缺失时 LLM 兜底。
把 CityKnowledgeDB.get_accommodation(city) 的结果作为 LLM prompt 的一个参考段落直接拼进去，明确标注"仅用于 analysis 字段的区域推荐理由，不得作为酒店白名单"。知识库没有该城市数据时，kb_hint 为空，LLM 基于坐标和自身知识写 analysis，零影响。改动只需在 AccommodationAgent.run() 里消费那个已经传入但被忽略的 knowledge_accommodation 参数。
所以我的建议是采用和 poi_enrich_node 类似但更轻量的模式：
把 CityKnowledgeDB.get_accommodation(city) 的结果作为 LLM prompt 的参考段落直接注入，用于指导 analysis 字段的区域推荐逻辑，但明确告诉 LLM "以下为区域参考信息，仅用于说明推荐区域的理由，options 中的酒店必须且只能来自 MCP 数据"。如果知识库里没有该城市的住宿指南，LLM 就自己基于地理坐标和通用知识来写 analysis，这就是你说的"兜底"。
具体改动很小，只需在 AccommodationAgent.run() 里把 knowledge_accommodation 消费起来：
python# run() 方法内，构建 prompt 之前
knowledge_accommodation = input_data.get("knowledge_accommodation", [])

#构建 prompt 时注入
kb_hint = ""
if knowledge_accommodation:
    kb_lines = "\n".join(f"  - {line}" for line in knowledge_accommodation)
    kb_hint = (
        f"\n【本地住宿区域参考（仅供 analysis 字段引用，不得作为酒店白名单）】\n"
        f"{kb_lines}\n"
        f"以上信息来自旅游攻略，用于辅助你判断'哪个区域值得推荐'，"
        f"但 options 中的酒店名称和价格必须来自上方 MCP 数据。\n"
    )
这样做的效果是：LLM 在写 analysis 时可以引用"上城区湖滨适合步行游西湖"这类本地知识来解释为什么推荐某个区域的酒店，但不会因此虚构酒店名或价格。如果知识库没有数据，kb_hint 为空字符串，LLM 正常走纯 MCP 数据分析，零影响。
